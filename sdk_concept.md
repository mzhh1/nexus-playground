星枢(Nexus)平台接入理念

以下是从子游戏 SDK 设计中提炼出的核心哲学与指导原则，用于指导星枢（Nexus）基础平台的架构设计与开发。

1. 核心理念：彻底的“关注点分离”

平台的核心职责是成为一个强大的“游戏运行时”（Game Runtime）。开发者则专注于提供两个独立的、可插拔的模块：“游戏核心逻辑”和“游戏交互界面”。

平台必须封装一切通用服务： 平台应全面接管所有与具体游戏规则无关的底层服务，包括但不限于：网络通信（WebSocket）、房间管理、玩家身份验证、状态持久化、数据同步以及 LLM API 的调用、路由和计费。

开发者的职责边界： 游戏开发者绝不应关心上述任何底层实现。他们只需聚焦于定义“游戏怎么玩”（逻辑）和“游戏长什么样”（界面）。

2. 理念一：游戏逻辑即“无状态服务”

平台必须将开发者提供的**“游戏核心逻辑”视为一个无状态的计算服务**（或一组纯函数）。

平台负责管理和持久化权威的游戏状态。

当需要进行游戏推演时，平台将当前状态（GameState）“喂”给游戏逻辑。

游戏逻辑负责计算并返回结果（例如，新的状态或一个错误）。

游戏逻辑本身绝不应持有或管理任何长效状态。这种设计保证了游戏的可复现性、可测试性和水平扩展能力。

3. 理念二：权威状态与玩家视角的严格分离

这是平台处理不完美信息和保障安全的核心。

平台必须维护一个权威的“上帝视角”状态（GameState）： 这是游戏的唯一真实数据源，包含所有玩家的私密信息（如所有人的手牌）。

平台绝不能将“上帝视角”状态泄露给任何客户端： 这是平台的安全红线。

平台必须依赖游戏逻辑生成“角色视角”（RolePerspective）： 平台的核心循环是：获取权威状态 -> 调用游戏逻辑，为其传入“角色A”的ID -> 游戏逻辑返回一个经过过滤的、只包含“角色A”应知信息的“角色视角” -> 平台再将这个“角色视角”发送给“角色A”对应的客户端。

4. 理念三：“角色视角”是驱动一切交互的核心媒介

“角色视角”是平台与所有玩家（无论人类还是 AI）沟通的唯一数据结构。平台必须利用这同一个结构实现双重驱动：

驱动 UI 渲染： 平台将“角色视角”推送给人类玩家的前端（Client）。前端 UI 是对这份数据的纯粹渲染。

驱动 LLM 决策： 当轮到 AI 玩家行动时，平台将完全相同的“角色视角”数据封装成一个结构化的 Prompt，发送给 LLM。

这种设计保证了人类和 AI 玩的是“同一个游戏”，并且为开发者提供了一个统一的（为 AI 和 UI）设计信息的接口。

5. 理念四：统一且唯一的“行动”接口

平台必须强制规定，所有对权威状态的修改尝试都必须通过一个标准化的“行动”（ActionJSON）接口来提交。

无论是人类玩家在 UI 上的点击，还是 LLM 玩家返回的决策JSON，都必须被平台转换为一个标准的“行动”对象。

平台随后将此“行动”连同当前“上帝视角”状态一起，提交给游戏逻辑的“行动处理函数”（processAction）进行验证和计算。

这是游戏状态变更的唯一途径，确保了所有操作的一致性和可验证性。

6. 理念五：声明式的“合法行动空间”

为了提升 AI 的准确性和优化 UI 的便捷性，平台必须支持一种“声明式”的行动模式。

平台应要求（或强烈建议）游戏逻辑在生成的“角色视角”中，包含一个当前所有合法行动的明确列表（action_space_definition）。

对 UI 的价值： 平台的前端 SDK 可以直接读取此列表来动态渲染可交互的按钮，开发者无需在前端重写复杂的“哪些按钮可点”的逻辑。

对 LLM 的价值： 平台可以将此列表作为强力约束（Constraint）注入到 Prompt 中，指示 LLM 必须从这些合法选项中进行选择，从而极大降低 LLM 产生非法或无效操作的概率。

7. 理念六：面向客户端的单向数据流与错误反馈

平台在设计客户端 SDK (@nexus-game/client) 时，必须遵循响应式和单向数据流的原则。

数据流： 平台推送视角 -> UI 渲染 -> 用户点击 -> 客户端提交行动 -> 平台处理 -> 平台推送新视角 ...

错误处理： 平台必须建立一个可靠的错误反馈闭环。当游戏逻辑验证一个“行动”并返回错误信息时（例如：“此处不能落子”），平台必须能将这个错误信息准确地推送回对应的客户端，以便 UI 及时向用户显示提示。
